# SRW J 한글화 이미지 통합 키트

## ★ 원클릭 적용 (권장)

```
python apply_all.py 입력.gba 출력.gba
```

한 번 실행으로 4가지를 모두 적용합니다:
1. 타이틀 화면(한글 로고 포함) — 타이틀로고/logo_snapshot.bin 이식
2. 시나리오 제목 68화 — 동봉 PNG 삽입 (titles_ko.txt 수정 후엔 `--render` 옵션)
3. 전투 메시지 70종 — 전투메시지/한국어 PNG 삽입
4. 부팅 저작권 화면 — credits_ko.txt 렌더·삽입
5. 인터페이스 4종(전투 UI 아이콘·MAP VIEW 리스트·경고 등) — 인터페이스/한국어 PNG 삽입

요구사항: Python 3 + Pillow(`pip install pillow`). scipy/numpy 불필요.
입력 ROM은 텍스트 패치가 적용된 SRW J ROM(그래픽 패치 적용 전)을 권장합니다.
모든 기록은 자동으로 라운드트립 검증되며, 슬롯 초과분은 0x1800000~ 에 재배치됩니다.

---

타이틀 화면 로고 → 시나리오 제목(68화) → 전투 메시지(70종)까지의
그래픽 자산 한글화 도구 모음. 모든 도구는 `공통/scn_title_lib.py`(ECD 코덱)를 사용.

## 폴더 구성

- 공통/scn_title_lib.py — ECD(LZSS) 디코더/최적파싱 인코더, 아카이브 입출력,
  IMG/SCR 해석, 슬롯 초과 시 ROM 끝(0x1800000~) 자동 재배치. 포맷 사양은 파일 상단 주석.
- 시나리오제목/ — 화 제목 일괄 생성 키트 (titles_ko.txt 수정 → 렌더 → 삽입)
- 전투메시지/ — 전투 메시지 이미지 교체 키트 (범용 IMG 교체기)

## 사용법 요약

스크립트 실행 시 공통 라이브러리가 import 가능해야 함:
`공통/scn_title_lib.py`를 실행 폴더에 복사하거나 PYTHONPATH에 추가.

### 1) 시나리오 제목 (화 0~67)
```
python make_titles.py titles_ko.txt 시나리오제목_한글     # 테이블 → PNG 렌더
python scn_title_insert.py in.gba out.gba 시나리오제목_한글 --grow
python scn_title_extract.py rom.gba 폴더 --sheet          # 검증용 재추출
```
- 화 n → IMG 3462+n, SCR 3531+n 페어. 삽입기가 SCR을 재생성하므로 배치 자유.
- 상세 규칙(캔버스/용량/번호 매핑)은 README_시나리오제목.md 참고.

### 2) 전투 메시지 (또는 임의 IMG 스트립)
```
python img_replace.py in.gba out.gba 한국어 일본어
```
- 파일명 img_NNNNN_*.png 로 자산 번호 인식. SCR 없는 단순 행우선 스트립 전용.
- 원본(일본어) PNG의 앞 16색 팔레트 기준 최근접 색 매핑,
  흰색으로 굳은 배경은 테두리 플러드필(허용오차 48)로 자동 투명화.
- 알파<128 = 색 0(투명). 인덱스/풀컬러 PNG 모두 입력 가능.

### 3) 타이틀 화면 로고
- 기존 전달본 `SRWJ_한글로고도구.zip`(make_korean_logo.py 등)을 그대로 사용.
- 로고 자산: IMG#15394(384타일) + SCR + PLT, ROM 끝 재배치 + 인덱스 갱신 방식.
- 본 키트의 공통 라이브러리와 같은 ECD 사양.


### 4) 부팅 저작권 화면 (라이선스 표기)
```
python credits_make.py in.gba out.gba credits_ko.txt        # 삽입
python credits_make.py --preview credits_ko.txt             # PNG 미리보기만
```
- 구조: IMG#15392(32x14 타일 풀) + SCR#15402(30x20 풀스크린 타일맵).
  줄 9~13은 줄 사이 틈 타일에 조각으로 패킹돼 있던 것을 SCR 재생성 방식으로 해결.
- credits_ko.txt = 13줄 텍스트 테이블(수정 가능). 글꼴 Galmuri9(SIL OFL, PIL에서 size=10).
- 줄 간격 12px 고정, 8px 격자 스냅으로 동일 문구(©, 반복 프리픽스) 타일 중복제거.


## 전투/UI 이미지 팔레트 정합 (p3979 그룹만 교정)

대부분의 전투/UI 이미지는 기존 추출/삽입이 정상입니다. **딱 하나의 예외**가
무기/파츠 라벨 그룹(파일명 p3979, 자산 15개: 3941~3956)으로, 이들의 실제 인게임
팔레트는 PLT#3899 뱅크2(노란 글자 + 검정 외곽선)입니다. 그래서 기존 추출본만 색이
어둡게(청록) 보였습니다.

교정 범위는 pal_override.json에 명시된 이 15개뿐이며, 그 외 자산은 오버라이드가 없어
**기존과 100% 동일하게** 동작합니다(파일명 PLT 뱅크0).

### 올바른 색으로 추출
```
python bm_extract.py 원본.gba 한국어  올바른색_추출본
```
- pal_override.json에 있는 자산만 교정 PLT로, 나머지는 파일명 PLT 뱅크0으로 추출.
- 출력은 16색 인덱스(P) PNG, 인덱스0 = 투명.
- `올바른색_추출본/` 폴더에 미리 추출된 결과가 있고, p3979 그룹은 노란색으로 나옵니다.

### 삽입 (기존 동작 보존)
```
python img_replace.py in.gba out.gba 한국어 일본어
```
- 기존 작업본(24bit/RGBA, 256색 등)은 예전과 **바이트 단위로 동일하게** 색매핑됩니다.
- 추가로, 한글 PNG를 16색 인덱스(P) 모드로 저장하면 인덱스를 그대로 보존하여 게임이
  실제 팔레트로 정확히 렌더합니다(색 변형 없음). p3979 그룹을 새로 칠할 때 권장.


### 노란색이 투명으로 나오던 문제 (해결됨)

올바른색 추출본(p3979 그룹)을 편집해 다시 넣었을 때 노란 글자가 투명으로 보이는 증상은,
편집기가 PNG를 24bit/RGBA로 저장하면서 삽입기가 "색매핑 폴백"을 타고, 그때 기준 팔레트가
어두운 청색(원본 추출본)이라 노란색의 최근접이 투명색(idx0)으로 잡혔기 때문이었습니다.

이제 삽입기가 다음을 자동 처리합니다:
- p3979 그룹은 ROM의 실제 교정 팔레트(PLT#3899 뱅크2)를 기준으로 삼습니다.
- 24bit/RGBA로 저장돼도 색이 16색 이하면 인덱스로 자동 환원합니다(인덱스환원 모드).
- 교정 대상에 한해 불투명 픽셀이 투명(idx0)으로 매핑되지 않도록 막습니다.
- 따라서 한글 PNG를 P모드/RGBA 어느 쪽으로 저장하든, 어떤 폴더를 기준으로 주든
  노란 글자가 보존됩니다. (p3979 외 자산의 삽입 결과는 종전과 100% 동일)

가장 안전한 방법은 여전히 **올바른색_추출본의 16색 팔레트를 유지한 채 글자만 바꿔
16색 인덱스(P) PNG로 저장**하는 것입니다.


### 글자가 투명/마젠타로 나오던 문제 (해결됨)

원인은 두 가지가 겹친 것이었습니다:
1. 편집 시 추출본을 16색 인덱스가 아닌 RGB(24bit)로 저장하면서 투명 배경이 검정으로
   평탄화됨 → idx0(투명)과 idx1(검정 외곽선)이 구분 불가능해짐.
2. 삽입기가 검정 배경을 투명화하려고 플러드필을 돌렸는데, 글자의 검정 외곽선이 배경과
   연결돼 있어 글자 속까지 투명으로 뚫렸음.

대응:
- 위험한 플러드필을 제거했습니다.
- 교정 자산(p3979)은 게임 팔레트의 투명색(idx0) 또는 마젠타(255,0,255)와 정확히 일치하는
  픽셀만 투명으로 보냅니다. 글자 외곽선(검정 idx1)은 색이 다르므로 절대 투명화되지 않습니다.

**가장 확실한 작업법(권장):** 올바른색_추출본의 PNG를 **16색 인덱스(P) 모드 그대로**
편집하세요(GIMP·Aseprite·Photoshop의 인덱스 모드). 그러면 마젠타 배경은 그대로 두고
글자만 바꿔도 인덱스가 보존돼 게임에서 정확한 색으로 나옵니다. RGB로 평탄화하지 마세요.
부득이 RGB로 저장하더라도, 배경 마젠타(255,0,255)와 글자색을 섞지(안티에일리어싱) 마세요.


## ECD 포맷 (검증 완료 사양)

- 인덱스: 0x1D6DE4, 4바이트 LE 상대 오프셋 x 15447개
- 헤더 16B: 'ECD\x01' + prefix길이(BE4) + 압축크기(BE4) + 원본크기(BE4)
  prefix(IMG/SCR/PLT 헤더 8B)는 비압축 저장, LZSS 윈도 미반영
- LZSS: 윈도 1024(0 초기화), 쓰기 시작 0x3BE. 플래그 LSB부터 비트1=리터럴,
  비트0=매치 2B: pos = b1|((b2>>6)<<8) (윈도 절대위치), len = (b2&0x3F)+3
- PLT는 ECD 아닌 원시 저장: 'PLT\0' + 색수 + BGR555 LE
- IMG: 'IMG\0' 0 tw th 0 + 4bpp 8x8 타일(행우선)
- SCR: 'SCR\0' 0 w h 0 + 셀 2B LE(타일 10비트, h/v플립, 뱅크)
- 시나리오 제목의 VRAM 변환: 저장타일 s ↔ VRAM (s//tw)*32 + (s%tw)
